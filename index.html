Case Study 01 · Western Union · 2023–2025

Reducing Card Payment
Abandonment by 18%

How reframing a payment failure problem as a decision system problem — and building a structured classification model — drove measurable results across 14+ European markets.

CompanyWestern Union
Markets14+ European
My RoleProduct Manager
Outcome18% Abandonment Reduction
01 · Context

The Environment

I owned card-enabled cross-border transaction flows across 14+ European markets within a high-volume payments platform at Western Union.

These transactions were high-value, time-sensitive, and highly dependent on issuer responses and fraud checks. Even small inefficiencies in handling failures directly impacted conversion, revenue realisation, and customer trust.

The Stakes
In cross-border payments at this scale, a failure isn't just a bad experience — it's lost revenue, lost trust, and a potential compliance risk. Every declined transaction is a decision point that the system and the user must navigate correctly.
02 · Problem

More Than a UX Problem

A significant portion of transactions were failing at the authorisation stage, leading to high abandonment rates. The surface-level reading was a UX problem. The reality was deeper.

1
Opaque issuer decline responses
Generic decline codes provided no actionable guidance on what to do next — retry, switch method, or stop.
2
Inconsistent retry behaviour
Without classification, retries were unpredictable — causing unnecessary attempts on hard declines and missed recoveries on soft ones.
3
Operational ambiguity
Frontline users lacked clarity on next best action — leading to inconsistent handling and avoidable drop-offs.
4
Fragmented handling across markets
Decline handling varied market to market with no standard logic — making the experience inconsistent and unscalable.
Net Effect
Users dropped off after failed attempts, resulting in lost revenue and a poor reliability perception — not because declines were unavoidable, but because the system offered no guidance on how to recover from them.
03 · My Approach

Reframing as a Decision System Problem

Instead of treating this as a UX issue alone, I approached it as a decision system problem. The question wasn't "how do we make the failure message better?" — it was "how do we make the system guide the right next action every time?"

1
Deep failure analysis
Analysed large-scale issuer decline datasets across top markets. Identified high-frequency decline categories, behavioural patterns, and segmented failures by recoverability and risk.
2
Core insight — not all declines are equal
The system needed to guide what to do next, not just show failure. This reframe changed everything about the solution direction.
3
Built a structured decline classification model
Partnered with Risk to design three tiers: Hard decline (block — do not retry) · Soft decline (safe retry with conditions) · Alternate path (suggest different payment method).
4
Embedded into product flows
Integrated classification logic into transaction workflows. Improved system guidance for retries and alternatives. Ensured consistency across all markets.
04 · Key Product Decisions

The Trade-offs I Navigated

1
Reduce unnecessary retries vs. maximise recovery
Higher retries could increase conversion but also risk fraud and issuer blocks. Decision: controlled retries based on decline type — not a blanket retry policy.
2
Standardisation vs. market-specific behaviour
Flexibility across markets risks inconsistency and unscalable logic. Decision: standard core classification with limited, well-defined market overrides.
3
System guidance vs. user autonomy
Over-automation removes human judgment; full autonomy creates inconsistency. Decision: guided recommendations embedded in the workflow — not forced actions.
05 · Impact

What We Achieved

18%
Reduction in card payment abandonment across high-value retail transactions
14+
European markets with consistent, structured decline classification
E2E
Full ownership from failure analysis through classification model, delivery, and optimisation
Beyond the Metric
This work established a scalable foundation for transaction decisioning — improving confidence in handling failures at scale and contributing to more predictable, reliable payment flows. The classification model itself became a reusable asset for future decline handling work.
06 · Reflection

What I Learned

The most valuable reframe on this project was recognising that decline handling wasn't a UX problem — it was a decision system problem. Solving it at the UX layer alone would have produced better error messages. Solving it at the system layer produced an 18% reduction in abandonment.

The data you don't have is as important as the data you do. Understanding which declines were recoverable — and which weren't — required going beyond surface metrics into the underlying issuer logic. That analysis was what made the classification model possible.

And finally: the trade-off between standardisation and flexibility is never fully resolved — it's managed. The decision to use standard core logic with limited market overrides was right at the time, and building in the ability to revisit it was just as important as the decision itself.

Case Study 02 · Cyber Resilience Platform · 2025

Improving Human-AI
Interaction

How I identified four critical UX gaps in an ML-powered ransomware detection platform and proposed a redesign grounded in Explainable AI principles.

IndustryCyber Security
Product TypeEnterprise AI Platform
My RoleProduct Manager
FocusExplainable AI (XAI)
01 · Context

The Product

I work as Product Manager on an enterprise cyber resilience platform that uses machine learning to detect ransomware attacks. The system compares file system snapshots across backup cycles, identifying attack patterns before they cause irreversible data loss.

The Core Challenge
The ML detection engine is highly accurate — but its outputs are not self-explanatory to the security admin who must act on them, often in the middle of the night during a live attack.
02 · How It Works

The Detection Pipeline

1
Snapshot Comparison
Compares today's backup against the previous one, tracking every file-level change across the system.
2
Signal Extraction
Generates statistical signals from the comparison — file behaviour patterns, size changes, and type distributions.
3
Anomaly Detection
ML model analyses signals to identify patterns characteristic of ransomware, such as mass file encryption or unusual deletions.
4
ML Model Prediction
A trained ML model combines all signals to predict whether an attack has occurred, with a confidence score indicating likelihood.
5
Alert Generation
Outputs alerts with severity levels and file-level detail into the application UI for security admin review.
03 · Problems Identified

Four Gaps I Found

Through close analysis of the live alert UI and deep understanding of the ML system at a product level, I independently identified four interconnected problems.

01
Alert Fatigue
Thousands of alerts across hundreds of pages. When everything is flagged at the same severity, labels become meaningless and admins face decision paralysis.
02
Explainability Gap
Raw ML scores displayed to security admins under pressure don't answer the only question that matters: should I recover this file right now?
03
Workflow Fragmentation
The user's job is to recover from an attack — but threat details and last clean backup are never shown together, forcing stressful navigation.
04
No Progressive Disclosure
Critical and High severity look nearly identical as text labels. No visual hierarchy pulls attention to what genuinely requires immediate action.
04 · User Journey

Before vs. After

The user's job-to-be-done isn't "read alerts" — it's "recover from a ransomware attack as fast as possible."

Current Journey — Broken
Open alerts → thousands of undifferentiated rows
Scroll through hundreds of pages to find real threat
Click alert → see raw ML score with no guidance
Navigate separately to find last clean backup
Make recovery decision with incomplete context
Proposed Journey — Fixed
Dashboard surfaces critical incidents with visual severity
Colour-coded indicators replace raw ML scores
Last clean backup shown inline with each alert
Action labels: "Recover Now" / "Investigate" / "Monitor"
Recovery decision in one screen, under 2 minutes
05 · Proposed Solutions

What I'd Build

Solution 1 — Colour-Coded ML Confidence Indicators

Replace raw model scores with a three-tier visual system mapped to recommended actions. The core principle of Explainable AI: don't show the model's math — show what the user should do.

Visual IndicatorML Confidence LevelRecommended Action
Critical
High confidence — attack highly likelyRecover immediately — show last clean backup inline
Suspicious
Moderate confidence — anomaly detectedInvestigate further before recovering
Clean
Low confidence — likely safeNo action needed — safe to use for recovery

Solution 2 — Unified Recovery Dashboard

Redesign the homepage to surface critical alerts grouped by attack event, the last confirmed clean backup per host, and a one-click recovery initiation — collapsing a multi-screen workflow into a single decision surface.

Solution 3 — Incident Grouping

A single attack affecting thousands of files should surface as one high-priority incident — not thousands of rows. This directly addresses alert fatigue and mirrors how leading security platforms present ML threat intelligence.

06 · Validation Plan

How I'd Test This

1
Usability Testing with Security Admins
Observe 3–5 enterprise admins using the current alert view during a simulated attack. Measure time-to-decision and error rate as baseline.
2
Define Success Metrics
Mean time to recovery decision · Alert dismissal rate · False positive acknowledgement rate · Admin confidence score (self-reported)
3
A/B Test Alert Grouping
Individual row view vs. incident-grouped view. Measure impact on response time and rate of missed critical alerts.

Multi-Agent Payment Decision System

End-to-End Payment Decision
Pipeline with Three Agents

An evolution from a single agent to a coordinated multi-agent system that simulates a real-world payment decision pipeline. The system chains three specialised agents through an orchestration layer — each with a distinct responsibility, designed to reflect how payments decisions actually work in production environments.

Why This Is Different
Most AI demos are single-step. Real payment systems are multi-step, conditional, and require explainable decisions at each stage. This system mirrors that reality — combining deterministic rule logic with LLM reasoning, and chaining agents based on outcomes rather than sequence.

The Three Agents

1
Fraud Detection Agent
Computes a risk score using deterministic logic and weighted signals — transaction amount, velocity, device trust, location anomaly, and behavioural patterns. Produces explainable outputs showing the contribution of each signal to the final score. No black box: every score is traceable.
2
Fraud Decision Agent
Combines rule-based thresholds with LLM reasoning to make a final decision under uncertainty. Outputs one of three decisions — approve, challenge (3DS), or block — along with a plain-English reasoning explanation and a confidence score. Designed to handle edge cases where rules alone are insufficient.
3
Routing Optimisation Agent
Runs only for approved transactions. Generates multiple routing options, scores each by balancing success rate against cost in basis points, and selects the best route with a fallback. Routing only happens when it's safe to — the orchestration layer ensures this.

The Orchestration Layer

The orchestration layer chains agents using conditional logic — not a fixed sequence. The decision tree is:

Block Path
Fraud Detection Agent scores transaction
Fraud Decision Agent outputs: Block
Pipeline stops — no routing
Unified output: blocked with reasoning
Approve Path
Fraud Detection Agent scores transaction
Fraud Decision Agent outputs: Approve
Routing Agent selects optimal route
Unified output: approved with route

Challenge decisions trigger a 3DS flow — the pipeline pauses for authentication before routing proceeds. The orchestration layer produces a unified structured output across all branches, making the system readable and auditable end-to-end.

Key Design Decisions

1
Hybrid architecture — deterministic + LLM
Rule-based scoring for the fraud signal computation (fast, auditable, consistent) combined with LLM reasoning for the final decision (handles nuance and edge cases). Neither alone is sufficient for a production payment system.
2
Explainability at every stage
Every agent produces human-readable reasoning alongside its structured output. The fraud score shows signal contributions. The decision agent explains why it approved, challenged, or blocked. The routing agent shows why it selected the chosen route. Explainability is a design requirement, not an afterthought.
3
Conditional chaining over sequential execution
The routing agent only runs when the decision agent approves. This mirrors how real payment systems work — downstream processes are gated by upstream decisions, not just triggered in sequence.
